======================================================================
ANTI-LEAKAGE RULES — MANDATORY AT EVERY STEP
======================================================================

These are NON-NEGOTIABLE. Violation = stop and fix before continuing.

L1 — StandardScaler: fit ONLY on training data. Never on full dataset.
L2 — HMM: train on train set only. Forward-predict on val/test.
L3 — Rolling features: use shift(1) or closed='left'. No centered windows.
L4 — Target variable: compute AFTER features. Never include future return.
L5 — Signal execution: signal at t → executed at OPEN of t+1.
L6 — Walk-forward gap: minimum L days gap between train end and val start.
L7 — MASI holidays: remove zero-volume days. Forward-fill max 2 days only.
L8 — GARCH proxy: re-estimate in each walk-forward window separately.

For every step:
- List which leakage rules are relevant
- Explain how each is mitigated
- Add assertion checks in code to catch violations programmatically

======================================================================
MOROCCAN MARKET CONSTRAINTS — ALWAYS APPLY
======================================================================

C1 — No VIX:
  → Build local volatility proxy (GARCH(1,1) preferred if ARCH effects confirmed)
  → Fallback: rolling realized volatility 21-day
  → Always justify chosen proxy statistically

C2 — Low liquidity:
  → Avoid overfitting assumptions
  → Use compact models (fewer parameters)
  → Treat zero-volume days as missing data

C3 — Limited observations (~2000-2500 daily obs):
  → Prefer models with < 10,000 parameters
  → Use early stopping aggressively (patience=5-10)
  → Prefer L=10 or L=20 input window over L=30
  → Test robustness across multiple window sizes

C4 — No tick data:
  → Daily frequency primary
  → Weekly frequency as robustness check

C5 — Missing data:
  → Document all gaps
  → Forward-fill max 2 consecutive days
  → Drop if gap > 5 consecutive days

C6 — Statistical significance difficult:
  → Never claim strong predictive power
  → Always report confidence intervals
  → Always report baseline comparison

======================================================================
MODEL PHILOSOPHY — MANDATORY
======================================================================

Deep learning is NOT automatically superior.

A complex model is justified ONLY IF it significantly outperforms:
  - Random Walk
  - Historical Mean
  - ARIMA
  - Random Forest

under ALL of these conditions simultaneously:
  - Walk-forward validation (not in-sample)
  - Realistic transaction costs (5-10 bps for MASI)
  - Leakage-free testing
  - Multiple evaluation windows (not just one test period)

If CNN-LSTM performance is:
  - unstable across windows → reject, propose Random Forest
  - statistically indistinguishable from ARIMA → reject, use ARIMA
  - overfitted (val >> test) → reject, simplify architecture
  - sensitive to random seed → report variance, reduce complexity

======================================================================
CNN-LSTM ARCHITECTURE CONSTRAINTS
======================================================================

Given ~2000-2500 MASI daily observations:

MAXIMUM allowed architecture:
  - Conv1D: 1 layer, max 32 filters, kernel_size=3
  - LSTM: 1 layer, max 32 units
  - Dropout: 0.2-0.3
  - Dense: 1 layer, max 16 units
  - Output: 1 neuron (regression)

Input window L: test L ∈ {10, 15, 20}. Do NOT default to L=30.
Features F: start with F ≤ 15. Add features only if validated.

Always justify:
  - Why Conv1D is useful for this specific data
  - Why LSTM captures what Conv1D cannot
  - Why HMM regime feature improves CNN-LSTM specifically

If not justified with validation data → simplify to LSTM-only.
